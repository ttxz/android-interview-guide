# Flutter 渲染原理与高级主题

## 一、Flutter 渲染引擎

### 1. 架构层次

```
Dart Framework（纯 Dart）
├── Material / Cupertino     → UI 组件库
├── Widgets                   → Widget 体系
├── Rendering                 → 布局和绘制
├── Animation / Gestures      → 动画和手势
└── Foundation                → 基础工具类

Flutter Engine（C++）
├── Skia                      → 2D 渲染引擎
├── Dart Runtime              → Dart VM
├── Text Layout (libtxt)      → 文字排版
└── Platform Channels          → 平台通信

Embedder（平台相关）
├── Android (Java/Kotlin)
├── iOS (Objective-C/Swift)
└── Web / Desktop
```

### 2. 渲染流水线

> **面试考点**：Flutter 如何渲染一帧？

```
VSync 信号 →
  1. Build Phase（Widget → Element → RenderObject）
  2. Layout Phase（布局计算，仅对标记 dirty 的节点）
  3. Paint Phase（绘制命令生成 Layer Tree）
  4. Composite Phase（合成图层）
  5. Rasterize Phase（Skia 光栅化 → GPU 渲染）
```

**关键点**：

- Flutter **自绘引擎**：不使用原生 View，直接通过 Skia 在 Canvas 上绘制
- 60fps / 120fps：每帧预算 16.6ms / 8.3ms
- 在独立线程上执行光栅化，避免阻塞 UI 线程

### 3. 三棵树详解

```dart
// Widget Tree：不可变的配置描述
// Element Tree：Widget 的实例，管理生命周期和子树更新
// RenderObject Tree：负责实际的布局（layout）和绘制（paint）

// 重建效率：
// Widget 重建 → Element 比较（canUpdate?）
//   → 相同类型 + 相同 key → 复用 Element，更新 RenderObject
//   → 不同 → 卸载旧 Element，创建新的

// canUpdate 判断
static bool canUpdate(Widget oldWidget, Widget newWidget) {
    return oldWidget.runtimeType == newWidget.runtimeType
        && oldWidget.key == newWidget.key;
}
```

### 4. 三棵树完整更新流程（面试核心）

> **面试必问**：`setState()` 调用后发生了什么？

```
setState(() { _count++; }) 调用后：

Step 1: 标记 Element 为 dirty
  State.setState()
    → Element.markNeedsBuild()
    → 加入 BuildOwner._dirtyElements 队列

Step 2: 等待下一帧（VSync 信号到来）
  BuildOwner.buildScope()
    → 遍历 _dirtyElements
    → 对每个 dirty Element 调用 rebuild()

Step 3: Widget 重构（Build Phase）
  Element.rebuild()
    → state.build(context) → 得到新 Widget 树
    → Element.updateChild() 对比新旧 Widget
        ├─ canUpdate? (runtimeType + key 相同)
        │   → true: 复用 Element，调用 element.update(newWidget)
        │            → RenderObject 更新属性（仅必要属性）
        └─ false: 卸载旧 Element，创建新 Element + RenderObject

Step 4: 布局（Layout Phase）
  RenderObject.markNeedsLayout()
    → 只对受影响的节点重新 layout（不是全树！）

Step 5: 绘制（Paint Phase）
  RenderObject.markNeedsPaint()
    → 生成绘制命令 Layer Tree

Step 6: 合成 + 光栅化（GPU 线程）
  Compositor → Skia → GPU
```

```dart
// 核心概念的生活类比：
// Widget = 建筑蓝图（每次都可以重新画，轻量）
// Element = 建筑施工队（长期存在，负责对比蓝图更新建筑）
// RenderObject = 真实建筑（重量级，尽量复用不拆）

// 这就是为什么 Flutter 鼓励频繁重建 Widget：
// Widget 重建开销极小（只是创建普通对象）
// 真正耗时的 RenderObject layout/paint 会被智能跳过
```

### 5. build() 触发时机汇总

```dart
// State.build() 会在以下情况被调用：
// 1. 首次插入 Widget 树（initState 之后）
// 2. 调用 setState()
// 3. 父 Widget 重建（父调用了 setState）—— ⚠️ 会导致不必要重建！
// 4. InheritedWidget 数据变化（dependOnInheritedWidgetOfExactType）
// 5. didUpdateWidget() 被调用（父 Widget 传入新配置）

// ✅ 优化：避免不必要的 build() 调用
// 方式1：const 构造函数（编译时常量，跳过重建）
const Text('不会重建的文字');

// 方式2：拆分 Widget（缩小 setState 影响范围）
// ❌ 整页 setState 导致所有子 Widget rebuild
class BadPage extends StatefulWidget {
  @override _BadPageState createState() => _BadPageState();
}
// ✅ 只把变化的部分抽成独立 StatefulWidget
class GoodPage extends StatelessWidget {
  @override Widget build(BuildContext context) {
    return Column(children: [
      LikeButton(),   // 只有这个内部 setState，不影响 GoodPage 其他子节点
      const ArticleContent(),
    ]);
  }
}

// 方式3：RepaintBoundary（隔离重绘范围）
RepaintBoundary(
  child: AnimatedWidget(),  // 频繁 repaint 但不影响父节点
)
```

### 2. 显式动画（Explicit）

```dart
class PulseAnimation extends StatefulWidget {
  @override
  State<PulseAnimation> createState() => _PulseAnimationState();
}

class _PulseAnimationState extends State<PulseAnimation>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;
  late Animation<double> _animation;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(
      duration: Duration(seconds: 1),
      vsync: this,  // TickerProvider，跟随屏幕刷新
    );
    _animation = Tween<double>(begin: 1.0, end: 1.5)
        .animate(CurvedAnimation(
            parent: _controller,
            curve: Curves.elasticOut,
        ));
    _controller.repeat(reverse: true);
  }

  @override
  Widget build(BuildContext context) {
    return AnimatedBuilder(
      animation: _animation,
      builder: (context, child) {
        return Transform.scale(scale: _animation.value, child: child);
      },
      child: Icon(Icons.favorite, size: 48, color: Colors.red),
    );
  }

  @override
  void dispose() {
    _controller.dispose();  // 必须释放！
    super.dispose();
  }
}
```

## 三、Flutter 与原生混合开发

### 1. 混合栈管理

```dart
// FlutterEngineGroup：多引擎方案
// 共享 GPU 上下文和 Dart Isolate 减少内存开销

// 单引擎方案（FlutterFragment / FlutterActivity）
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        val flutterFragment = FlutterFragment
            .withCachedEngine("my_engine")
            .build<FlutterFragment>()

        supportFragmentManager.commit {
            replace(R.id.container, flutterFragment)
        }
    }
}
```

### 2. 预热引擎

```kotlin
// Application 中预热 Flutter 引擎
class MyApp : Application() {
    lateinit var flutterEngine: FlutterEngine

    override fun onCreate() {
        super.onCreate()
        flutterEngine = FlutterEngine(this)
        flutterEngine.dartExecutor.executeDartEntrypoint(
            DartExecutor.DartEntrypoint.createDefault()
        )
        FlutterEngineCache.getInstance().put("my_engine", flutterEngine)
    }
}
```

### 3. 混合开发方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| FlutterActivity/Fragment | 官方支持 | 每次打开都要初始化 |
| 预热引擎（缓存） | 秒开 | 内存占用增加 |
| FlutterEngineGroup | 多引擎共享资源 | API 复杂 |
| FlutterBoost（闲鱼） | 完整的混合栈管理 | 第三方维护 |
