## 一、事件分发机制 — 彻底搞懂

### 🎯 一句话总结
>
> 事件从 Activity → ViewGroup → View **自上而下传递**，如果没人消费，再**自下而上回传**，形成一个 **U 型**。

### 📖 生活类比 — 公司审批系统

```
老板（Activity）  收到一个任务
  │
  ├→ 问经理（ViewGroup）："你要亲自处理吗？"
  │   └→ 经理说"不拦截"（onInterceptTouchEvent = false）
  │       │
  │       ├→ 把任务交给员工（View）
  │       │   └→ 员工说"我来做"（onTouchEvent = true）
  │       │       → ✅ 搞定！后续任务都直接给这个员工
  │       │
  │       │   └→ 员工说"我不会"（onTouchEvent = false）
  │       │       → 退回给经理处理
  │       │       → 经理也不会？→ 退回给老板
  │       │
  │   └→ 经理说"我拦截"（onInterceptTouchEvent = true）
  │       → 经理自己处理（经理的 onTouchEvent）
```

### 🔬 完整流程详解

```
一个手指按下（ACTION_DOWN）：

Activity.dispatchTouchEvent(ev)
  │
  │  "Activity 把事件传给窗口"
  ↓
ViewGroup.dispatchTouchEvent(ev)
  │
  │  先问自己："我要不要拦截？"
  ↓
ViewGroup.onInterceptTouchEvent(ev) ─── true → ViewGroup 自己消费
  │                                              ↓
  │ false（不拦截）                      ViewGroup.onTouchEvent(ev)
  ↓
ChildView.dispatchTouchEvent(ev)
  │
  │  View 没有 onInterceptTouchEvent（它是终端）
  ↓
ChildView.onTouchEvent(ev) ──── true → ✅ View 消费了事件
  │                                    后续 MOVE/UP 都给它
  │ false（不消费）
  ↓
ViewGroup.onTouchEvent(ev) ──── true → ViewGroup 消费
  │
  │ false
  ↓
Activity.onTouchEvent(ev) ──── 最后兜底处理
```

### ⚡ 最关键的三条规则

```
规则1：DOWN 事件决定一切
  └→ 谁消费了 DOWN，后续 MOVE/UP 都归它
  └→ 如果没人消费 DOWN，后续事件直接给 Activity

规则2：拦截是不可逆的
  └→ ViewGroup 一旦拦截，后续事件不再问 onInterceptTouchEvent
  └→ 子 View 会收到 ACTION_CANCEL

规则3：子 View 可以"反抗"
  └→ requestDisallowInterceptTouchEvent(true)
  └→ 禁止父 ViewGroup 拦截（但对 DOWN 事件无效）
```

### 🛠️ 滑动冲突实战

**场景**：ViewPager（水平滑动）嵌套 RecyclerView（垂直滑动）

```kotlin
// 外部拦截法（在父 View 决定）
class SmartViewPager : ViewPager {
    private var startX = 0f
    private var startY = 0f

    override fun onInterceptTouchEvent(ev: MotionEvent): Boolean {
        when (ev.action) {
            ACTION_DOWN -> {
                startX = ev.x
                startY = ev.y
                return false  // DOWN 不拦截！（规则1）
            }
            ACTION_MOVE -> {
                val dx = abs(ev.x - startX)  // 水平距离
                val dy = abs(ev.y - startY)  // 垂直距离
                return dx > dy  // 水平方向大 → 我来处理（ViewPager滑动）
                                // 垂直方向大 → 交给子View（RecyclerView滚动）
            }
        }
        return super.onInterceptTouchEvent(ev)
    }
}
```

### 💡 面试回答模板

> "事件分发是一个 U 型流程：DOWN 事件从 Activity 传到 ViewGroup 再传到 View，如果 View 不消费则逐级回传给父级处理。ViewGroup 通过 `onInterceptTouchEvent` 决定是否拦截。关键规则是：DOWN 事件的消费者决定后续事件归属；子 View 可通过 `requestDisallowInterceptTouchEvent` 阻止父级拦截。解决滑动冲突通常用外部拦截法——在父 View 的 `onInterceptTouchEvent` 中根据滑动方向判断。"

## 二、Handler 消息机制 — 为什么 Looper 死循环不卡？

### 📖 生活类比 — 快递驿站

```
Looper = 驿站值班员（一个线程一个值班员）
MessageQueue = 快递架子（按送达时间排好的队列）
Handler = 寄件人/收件人（既能寄又能取）
Message = 快递包裹

工作流程：
1. Handler 往架子上放快递（sendMessage）
2. 值班员不停检查架子（Looper.loop）
3. 有快递到时间了 → 按照面单上的收件人（msg.target）通知取件
4. 没有快递 → 值班员打盹（epoll_wait 阻塞），不消耗体力

为什么值班员"死循环"不累？
→ 因为没活干的时候他在 **睡觉**（阻塞），不是空转！
→ 有新快递来了（新消息入队），会 **叫醒** 他（nativeWake）
```

### 🔬 epoll 阻塞的本质

```
主线程 Looper.loop()
  │
  for(;;) {
  │   msg = queue.next()
  │       │
  │       ↓
  │   队列为空？──→ 调用 nativePollOnce(-1)
  │       │              │
  │       │         底层是 epoll_wait()
  │       │         线程状态：SLEEPING 💤
  │       │         CPU 使用率：0%
  │       │              │
  │       │         有新消息 enqueue → nativeWake()
  │       │         epoll_wait 返回 → 线程醒来 ⏰
  │       │              │
  │       ↓              ↓
  │   取出消息 msg
  │   msg.target.dispatchMessage(msg) ← 这才是真正执行代码的地方
  │       │
  │   如果这一步耗时 > 5秒 → ANR！❌
  │       │
  │   msg.recycle() → 回收到对象池
  }
```

### ⚡ 关键结论

```
1. Looper 死循环 ≠ ANR
   ANR = 消息处理太慢（handleMessage 超时）
   死循环 = 等待消息，没消息时休眠

2. 所有 UI 操作都是 Message
   点击事件   → Message
   生命周期   → Message (H.LAUNCH_ACTIVITY 等)
   View 绘制  → Message (Choreographer)
   延时任务   → Message (postDelayed)

3. 主线程的一切都在 Handler 中执行
   所以"不要在主线程做耗时操作"
   = "不要在 handleMessage 里运行太久"
```

## 三、RecyclerView 四级缓存 — 流程图解

### 📖 生活类比 — 图书馆借还系统

```
一级 Scrap   = 你手上正在看的书（屏幕上可见的 ViewHolder）
               → 刷新时暂存，不需要重新找

二级 Cache   = 你桌上刚放下的书（刚滑出屏幕的 2 个 ViewHolder）
               → 拿回来不需要重新翻页（不需要 bind）

三级 Extension = 你的私人藏书（自定义缓存，很少用）

四级 Pool    = 图书馆书架（按类别分，每类最多 5 本）
               → 只知道是什么类型的书，要重新翻到对应页（需要 bind）

五级 创建    = 买新书（createViewHolder + bindViewHolder）
```

### 🔬 滑动时缓存查找流程

```
用户手指向上滑 → Item 15 即将显示在屏幕底部

Step 1: 查 Scrap（屏幕内暂存）
  └→ Item 15 在屏幕内吗？→ No
  
Step 2: 查 CachedViews（最近移出的 2 个）
  └→ 存的是 Item 6, Item 7（刚从顶部滑出）
  └→ 按 position 匹配：都不是 15 → Miss

Step 3: 查 ViewCacheExtension（自定义）
  └→ 通常没有 → Skip

Step 4: 查 RecycledViewPool（按 viewType）
  └→ viewType=0 的池子里有回收的 ViewHolder → Hit! ✅
  └→ 但是！需要重新 bindViewHolder（因为数据不对）

Step 5: 都没有
  └→ createViewHolder() + bindViewHolder() → 最贵的操作 💰
```

### ⚡ 性能优化核心

```
目标：让 bindViewHolder 和 createViewHolder 尽量少调用

方法：
1. 增大 CachedViews → recyclerView.setItemViewCacheSize(10)
   → 更多刚滑出的 ViewHolder 可以不 bind 直接复用

2. 共享 Pool → 多个相同 viewType 的列表共享
   → 减少 create 次数

3. setHasFixedSize(true) → 减少不必要的 layout 计算

4. DiffUtil 精确更新 → 减少不必要的 bind

5. Payload 局部更新 → 只更新变化的部分，不重新 bind 整个 ViewHolder
```

## 四、Binder 一次拷贝 — mmap 原理图解

### 📖 生活类比

**传统IPC（管道/Socket）= 寄快递**：

```
发送方 → 复制到快递包 → 快递到驿站 → 复制到收件人手中
       第1次拷贝                   第2次拷贝
```

**Binder = 共享白板**：

```
发送方 → 写在共享白板上 → 收件人直接看白板
       第1次拷贝           不需要拷贝（mmap直接映射到内存）
```

### 🔬 mmap 内存映射

```
App 进程 A (用户空间)           App 进程 B (用户空间)
┌──────────────────┐          ┌──────────────────┐
│                  │          │   ┌────────────┐  │
│  数据 "Hello"    │          │   │ mmap 映射区 │  │
│       │          │          │   │            │  │
└───────│──────────┘          │   │  可直接读  │  │
        │                     │   │            │  │
        │  copy_from_user     └───│────────────│──┘
        │  (第1次也是唯一一次)       │            │
        ↓                         │            │
┌───────────────────────────────────────────────┐
│              内核空间                          │
│   ┌────────────────────────────────────┐      │
│   │  Binder 驱动的内核缓冲区           │      │
│   │  与 B 的用户空间 mmap 映射同一块   │←─────┘
│   │  物理内存                          │  (mmap)
│   └────────────────────────────────────┘
└───────────────────────────────────────────────┘

数据流：A → copy_from_user → 内核缓冲区 ←mmap→ B用户空间
        只拷贝了1次！B 通过 mmap 直接读内核中的数据
```

### 💡 面试回答模板

> "Binder 之所以只需要一次数据拷贝，是因为接收端进程通过 `mmap()` 将自己的一块用户空间地址映射到内核的 Binder 缓冲区。发送端数据通过 `copy_from_user` 拷贝到内核缓冲区（第1次拷贝），而接收端通过 mmap 映射直接访问这块内核缓冲区的数据，无需第2次拷贝。相比传统 IPC 的两次拷贝，Binder 减少了一半的内存拷贝开销。"

## 五、ViewModel 存活原理

### 📖 生活类比

```
Activity = 你的办公室（配置变更时要搬家/装修）
ViewModel = 你的保险柜（搬家时保险柜不动）

搬家流程：
1. 旧办公室关门 → Activity.onDestroy()
   但保险柜放到走廊 → ViewModel 存入 NonConfigurationInstances

2. 新办公室开门 → 新 Activity.onCreate()
   保险柜搬进新办公室 → 从 NonConfigurationInstances 取出 ViewModel

3. 真正退休了（finish()） →保险柜也销毁 → ViewModel.onCleared()
```

### 🔬 源码关键路径

```kotlin
// 1. Activity 销毁前保存 ViewModel
// ActivityThread.performDestroyActivity()
//   → activity.retainNonConfigurationInstances()
//     → 把 ViewModelStore 存到 NonConfigurationInstances 中

// 2. 新 Activity 创建时恢复
// ActivityThread.performLaunchActivity()
//   → activity.attach(... lastNonConfigurationInstances ...)
//     → ViewModelStore 从 NonConfigurationInstances 中取出

// 3. 判断是否是配置变更（而非 finish）
if (activity.isChangingConfigurations) {
    // 保留 ViewModelStore → ViewModel 存活
} else {
    // 调用 viewModelStore.clear() → ViewModel.onCleared()
}
```

### ⚡ 关键结论

```
ViewModel 存活的条件：
  ✅ 横竖屏切换（配置变更） → 存活
  ✅ 语言切换、字体大小等变更 → 存活
  ❌ 用户按返回键 → 销毁（onCleared）
  ❌ finish() → 销毁
  ❌ 进程被杀 → 销毁（需配合 SavedStateHandle）
```
