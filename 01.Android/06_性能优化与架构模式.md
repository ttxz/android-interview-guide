## 一、性能优化

### 1. 布局优化

> **面试常问**：如何减少过度绘制和布局层级？

```xml
<!-- 1. 使用 ConstraintLayout 减少层级 -->
<androidx.constraintlayout.widget.ConstraintLayout>
    <!-- 扁平化布局，避免多层嵌套 -->
</androidx.constraintlayout.widget.ConstraintLayout>

<!-- 2. merge 标签：减少一层 ViewGroup -->
<merge xmlns:android="http://schemas.android.com/apk/res/android">
    <TextView ... />
    <ImageView ... />
</merge>

<!-- 3. ViewStub：延迟加载不常用布局 -->
<ViewStub
    android:id="@+id/stub_error"
    android:inflatedId="@+id/error_layout"
    android:layout="@layout/layout_error"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />
```

### 2. 内存优化

#### Bitmap 优化

```kotlin
// 1. 采样压缩
fun decodeSampledBitmap(res: Resources, resId: Int, reqWidth: Int, reqHeight: Int): Bitmap {
    val options = BitmapFactory.Options().apply {
        inJustDecodeBounds = true  // 只读尺寸
    }
    BitmapFactory.decodeResource(res, resId, options)

    options.inSampleSize = calculateInSampleSize(options, reqWidth, reqHeight)
    options.inJustDecodeBounds = false
    return BitmapFactory.decodeResource(res, resId, options)
}

// 2. 使用合适的像素格式
options.inPreferredConfig = Bitmap.Config.RGB_565  // 不需要透明通道，内存减半

// 3. Bitmap 复用
options.inBitmap = existingBitmap  // 复用已有 Bitmap 内存
```

#### 内存泄漏

```kotlin
// 常见泄漏场景：
// 1. 非静态内部类/匿名内部类持有外部类引用
// ❌ 错误
class MyActivity : Activity() {
    val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            // 持有 Activity 引用
        }
    }
}

// ✅ 正确：使用静态内部类 + 弱引用
class MyActivity : Activity() {
    class MyHandler(activity: MyActivity) : Handler(Looper.getMainLooper()) {
        private val activityRef = WeakReference(activity)
        override fun handleMessage(msg: Message) {
            activityRef.get()?.handleMsg(msg)
        }
    }
}

// 2. 单例持有 Activity Context
// ❌ 错误
object Manager {
    lateinit var context: Context  // 如果传入 Activity，泄漏！
}
// ✅ 使用 ApplicationContext

// 3. 未取消的回调和监听器
// 在 onDestroy() 中取消注册
```

### 3. ANR 分析

> **面试常问**：什么情况会导致 ANR？如何排查？

**ANR 触发条件**：

| 类型 | 超时时间 |
|------|---------|
| 输入事件（触摸/按键） | 5 秒 |
| BroadcastReceiver | 前台 10 秒 / 后台 60 秒 |
| Service | 前台 20 秒 / 后台 200 秒 |
| ContentProvider | 10 秒 |

**排查方法**：

1. `adb bugreport` 获取 ANR traces
2. 查看 `/data/anr/traces.txt`
3. 使用 StrictMode 检测主线程操作
4. Android Profiler CPU 分析

```kotlin
// StrictMode 检测
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectDiskReads()
            .detectDiskWrites()
            .detectNetwork()
            .penaltyLog()
            .build()
    )
}
```

## 二、架构模式

### 1. MVC → MVP → MVVM → MVI 演进

#### MVC（Model-View-Controller）

```
View（XML） ←→ Controller（Activity/Fragment） ←→ Model
问题：Activity 同时承担 View 和 Controller 职责，过于臃肿
```

#### MVP（Model-View-Presenter）

```kotlin
// View 接口
interface UserView {
    fun showLoading()
    fun showUsers(users: List<User>)
    fun showError(message: String)
}

// Presenter
class UserPresenter(private val view: UserView, private val repo: UserRepository) {
    fun loadUsers() {
        view.showLoading()
        repo.getUsers(
            onSuccess = { users -> view.showUsers(users) },
            onError = { msg -> view.showError(msg) }
        )
    }
}

// Activity 实现 View 接口
class UserActivity : AppCompatActivity(), UserView {
    private val presenter = UserPresenter(this, UserRepository())

    override fun showLoading() { /* ... */ }
    override fun showUsers(users: List<User>) { /* ... */ }
    override fun showError(message: String) { /* ... */ }
}
```

- **优点**：View 和 Presenter 分离
- **缺点**：接口爆炸、Presenter 持有 View 引用（内存泄漏风险）

#### MVVM（Model-View-ViewModel）

```kotlin
// ViewModel
class UserViewModel(private val repo: UserRepository) : ViewModel() {
    private val _uiState = MutableStateFlow<UiState>(UiState.Loading)
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun loadUsers() {
        viewModelScope.launch {
            _uiState.value = UiState.Loading
            try {
                val users = repo.getUsers()
                _uiState.value = UiState.Success(users)
            } catch (e: Exception) {
                _uiState.value = UiState.Error(e.message ?: "未知错误")
            }
        }
    }
}

// View（Activity/Fragment）
class UserFragment : Fragment() {
    private val viewModel: UserViewModel by viewModels()

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.uiState.collect { state ->
                    when (state) {
                        is UiState.Loading -> showLoading()
                        is UiState.Success -> showData(state.data)
                        is UiState.Error -> showError(state.message)
                    }
                }
            }
        }
    }
}
```

- **优点**：数据驱动、生命周期感知、无 View 引用
- **缺点**：复杂状态管理困难

#### MVI（Model-View-Intent）

```kotlin
// State：不可变，描述 UI 完整状态
data class UserState(
    val isLoading: Boolean = false,
    val users: List<User> = emptyList(),
    val error: String? = null
)

// Intent：用户意图
sealed class UserIntent {
    object LoadUsers : UserIntent()
    data class DeleteUser(val userId: Long) : UserIntent()
    data class SearchUsers(val query: String) : UserIntent()
}

// ViewModel
class UserViewModel : ViewModel() {
    private val _state = MutableStateFlow(UserState())
    val state: StateFlow<UserState> = _state.asStateFlow()

    fun processIntent(intent: UserIntent) {
        when (intent) {
            is UserIntent.LoadUsers -> loadUsers()
            is UserIntent.DeleteUser -> deleteUser(intent.userId)
            is UserIntent.SearchUsers -> searchUsers(intent.query)
        }
    }

    private fun loadUsers() {
        viewModelScope.launch {
            _state.update { it.copy(isLoading = true) }
            try {
                val users = repo.getUsers()
                _state.update { it.copy(isLoading = false, users = users, error = null) }
            } catch (e: Exception) {
                _state.update { it.copy(isLoading = false, error = e.message) }
            }
        }
    }
}
```

- **优点**：单向数据流、状态可预测、易测试
- **缺点**：模板代码多

### 2. 架构模式对比

| 特性 | MVC | MVP | MVVM | MVI |
|------|-----|-----|------|-----|
| 数据流 | 双向 | 双向 | 数据绑定 | 单向 |
| View 与逻辑 | 耦合 | 接口分离 | 观察者 | 单向驱动 |
| 可测试性 | 差 | 良好 | 良好 | 优秀 |
| 复杂度 | 低 | 中 | 中 | 高 |
| 状态管理 | 分散 | Presenter | ViewModel | 集中式 |

### 3. Clean Architecture

```
Presentation Layer (UI)
    ↓
Domain Layer (Use Cases, 纯 Kotlin)
    ↓
Data Layer (Repository, DataSource)
```

```kotlin
// Domain Layer - Use Case
class GetUsersUseCase(private val repository: UserRepository) {
    suspend operator fun invoke(): Result<List<User>> {
        return repository.getUsers()
    }
}

// Domain Layer - Repository 接口
interface UserRepository {
    suspend fun getUsers(): Result<List<User>>
}

// Data Layer - Repository 实现
class UserRepositoryImpl(
    private val remoteDataSource: RemoteDataSource,
    private val localDataSource: LocalDataSource
) : UserRepository {
    override suspend fun getUsers(): Result<List<User>> {
        return try {
            val users = remoteDataSource.fetchUsers()
            localDataSource.saveUsers(users)
            Result.success(users)
        } catch (e: Exception) {
            val cached = localDataSource.getUsers()
            if (cached.isNotEmpty()) Result.success(cached)
            else Result.failure(e)
        }
    }
}
```
