# Android 四大组件与跨进程通信

## 一、Service

### 1. Service 生命周期

```
启动服务（startService）：
onCreate() → onStartCommand() → [运行中] → onDestroy()

绑定服务（bindService）：
onCreate() → onBind() → [绑定中] → onUnbind() → onDestroy()

混合模式（同时启动 + 绑定）：
onCreate() → onStartCommand() + onBind() → ... → onUnbind() → onDestroy()
// 需要同时 stopService + unbindService 才会销毁
```

### 2. startService vs bindService

| 特性 | `startService` | `bindService` |
|------|---------------|---------------|
| 生命周期 | 独立于调用者 | 随绑定者销毁 |
| 通信 | 单向（Intent 传参） | 双向（IBinder） |
| 回调 | `onStartCommand()` | `onBind()` |
| 停止 | `stopSelf()` / `stopService()` | 所有绑定断开后自动销毁 |

### 3. 前台服务（Android 8.0+）

```kotlin
class DownloadService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = NotificationCompat.Builder(this, CHANNEL_ID)
            .setContentTitle("下载中")
            .setContentText("正在下载文件...")
            .setSmallIcon(R.drawable.ic_download)
            .build()

        startForeground(NOTIFICATION_ID, notification)  // 必须在 5 秒内调用
        // 执行下载...
        return START_NOT_STICKY
    }

    override fun onBind(intent: Intent?): IBinder? = null
}

// onStartCommand 返回值
START_NOT_STICKY     // 被杀后不重建
START_STICKY         // 被杀后重建，intent 为 null
START_REDELIVER_INTENT // 被杀后重建，重传最后的 intent
```

### 4. Android 12+ 前台服务限制

```xml
<!-- 必须声明前台服务类型 -->
<service
    android:name=".LocationService"
    android:foregroundServiceType="location" />
```

## 二、BroadcastReceiver

### 1. 静态注册 vs 动态注册

| 特性 | 静态注册（Manifest） | 动态注册（代码） |
|------|---------------------|-----------------|
| 注册方式 | AndroidManifest.xml | `registerReceiver()` |
| 生命周期 | 应用安装后常驻 | 随注册组件生命周期 |
| Android 8.0+ | 多数隐式广播被限制 | 不受限制 |

```kotlin
// 动态注册
val receiver = object : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val isConnected = intent.getBooleanExtra(
            ConnectivityManager.EXTRA_NO_CONNECTIVITY, false
        )
    }
}
registerReceiver(receiver, IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION))

// 必须在适当时机取消注册！
override fun onDestroy() {
    unregisterReceiver(receiver)
    super.onDestroy()
}
```

### 2. 有序广播 vs 无序广播

```kotlin
// 无序广播：所有接收者几乎同时收到
sendBroadcast(intent)

// 有序广播：按优先级顺序传递，可拦截
sendOrderedBroadcast(intent, null)
// 接收者中可以 abortBroadcast() 阻拦

// 本地广播（已废弃，推荐用 LiveData/Flow 替代）
LocalBroadcastManager.getInstance(this).sendBroadcast(intent)
```

## 三、ContentProvider

> **面试常问**：ContentProvider 的启动时机

```kotlin
class MyContentProvider : ContentProvider() {
    // 在 Application.onCreate() 之前创建！
    override fun onCreate(): Boolean {
        // 初始化数据库等
        return true
    }

    override fun query(uri: Uri, ...): Cursor? { /* ... */ }
    override fun insert(uri: Uri, values: ContentValues?): Uri? { /* ... */ }
    override fun update(uri: Uri, values: ContentValues?, ...): Int { /* ... */ }
    override fun delete(uri: Uri, ...): Int { /* ... */ }
    override fun getType(uri: Uri): String? { /* MIME 类型 */ }

```

**ContentProvider 的作用**：

1. 跨进程数据共享的标准接口
2. 与系统功能集成（联系人、媒体库）
3. 第三方库的自动初始化（App Startup 的底层机制）

## 四、Binder 机制

> **面试核心考点**：Android IPC 的基石

### 1. 为什么使用 Binder？

| IPC 方式 | 次数拷贝 | 特点 |
|---------|---------|------|
| 共享内存 | 0 | 复杂、需同步 |
| **Binder** | **1** | 安全、高效 |
| Socket | 2 | 开销大 |
| 管道 | 2 | 单向 |

**Binder 优势**：

- 一次数据拷贝（mmap 内存映射）
- C/S 架构，接口清晰
- 可获取调用方 UID/PID（安全性）

### 2. Binder 通信流程

```
Client 进程          Binder 驱动          Server 进程
    │                    │                    │
    │──transact()──→    │                    │
    │                    │──copy_from_user──→ │
    │                    │    (mmap共享)       │
    │                    │                    │──onTransact()
    │                    │←─── reply data ───│
    │←── 返回结果 ──────│                    │
```

### 3. AIDL 使用

```aidl
// IBookManager.aidl
interface IBookManager {
    List<Book> getBookList();
    void addBook(in Book book);
    void registerCallback(IOnBookChangedListener listener);
}
```

```kotlin
// Server 端
class BookService : Service() {
    private val books = mutableListOf<Book>()

    private val binder = object : IBookManager.Stub() {
        override fun getBookList(): List<Book> = books
        override fun addBook(book: Book) {
            books.add(book)
        }
    }

    override fun onBind(intent: Intent): IBinder = binder
}

// Client 端
val connection = object : ServiceConnection {
    override fun onServiceConnected(name: ComponentName, service: IBinder) {
        val bookManager = IBookManager.Stub.asInterface(service)
        val books = bookManager.bookList  // 跨进程调用
    }

    override fun onServiceDisconnected(name: ComponentName) {}
}

bindService(
    Intent().setComponent(ComponentName("com.server", "com.server.BookService")),
    connection,
    Context.BIND_AUTO_CREATE
)
```

### 4. Messenger（轻量 IPC）

```kotlin
// 基于 Handler 的简单 IPC（串行处理消息）
// Server
class MessengerService : Service() {
    val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            when (msg.what) {
                MSG_SAY_HELLO -> {
                    // 回复客户端
                    val reply = Message.obtain(null, MSG_REPLY)
                    msg.replyTo.send(reply)
                }
            }
        }
    }
    override fun onBind(intent: Intent) = Messenger(handler).binder
}
```

## 五、其他 IPC 方式对比

| 方式 | 适用场景 | 特点 |
|------|---------|------|
| **AIDL** | 自定义复杂接口、并发调用 | 功能最强、编码复杂 |
| **Messenger** | 简单消息、串行通信 | 基于 Handler，简单 |
| **ContentProvider** | 数据共享（CRUD） | 标准化接口 |
| **Bundle** | 四大组件间传数据 | 最简单，有大小限制 |
| **文件共享** | 简单数据 | 并发问题 |
| **Socket** | 网络通信、跨设备 | 灵活但开销大 |
