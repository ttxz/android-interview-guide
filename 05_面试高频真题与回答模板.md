# Android 面试高频真题与回答模板

> 精选 10 年经验级别 Android 高级工程师最常被问到的真题，附上**标准回答模板**和**面试官想听的关键词**。

## 一、Kotlin 篇

### Q1：说说 `val` 和 `const val` 的区别？

**🎯 面试官想听的关键词**：编译时常量、字节码内联、运行时赋值

> **回答**：`val` 是运行时不可变变量，值在运行时确定，可以是函数调用的结果（如 `val now = Date()`）。`const val` 是编译时常量，值必须是基本类型或 String，在编译时直接内联到调用处（类似 Java 的 `static final` 字段被内联）。反编译字节码可以看到，`const val` 的引用直接被替换为字面值。

---

### Q2：`lateinit` 和 `by lazy` 怎么选？

**🎯 关键词**：var vs val、线程安全、原始类型

> **回答**：两者都是延迟初始化，但有本质区别。`lateinit` 用于 `var`，不支持基本类型和可空类型，适合依赖注入等场景（如 Dagger/Hilt 注入的属性）；初始化后可以修改值。`by lazy` 用于 `val`，默认线程安全（SYNCHRONIZED 模式），第一次访问时初始化且只执行一次，适合重量级对象的懒加载。日常开发中，如果是 val 就用 `by lazy`，如果需要可变或注入就用 `lateinit`。

---

### Q3：协程和线程的本质区别？

**🎯 关键词**：用户态调度、CPS变换、不阻塞线程

> **回答**：线程是操作系统层面的调度单位，每个线程约占 1MB 栈空间，切换需要内核态/用户态切换。协程是编译器层面的抽象，本质是 CPS 变换后的状态机对象，只占几十字节。协程的挂起不会阻塞线程——通过 `Continuation` 把"后续代码"打包，让出线程去做其他事，异步操作完成后通过 `resumeWith` 恢复执行。同一个线程可以运行成千上万个协程，而创建同等数量级的线程会导致 OOM。实际开发中，协程配合 `Dispatchers.IO` 和 `Dispatchers.Main` 可以优雅地完成"IO 线程请求数据、主线程更新 UI"的场景。

---

### Q4：Flow 和 LiveData 该怎么选？

**🎯 关键词**：生命周期感知、操作符丰富、多平台

> **回答**：如果是纯 Android 项目且需求简单，LiveData 够用——它自动感知生命周期，不需要手动管理订阅。但 Flow 是更好的选择：操作符丰富（map、filter、debounce、combine 等），支持 Kotlin 多平台，且 `StateFlow` 可以完全替代 LiveData。推荐做法是 ViewModel 中用 `StateFlow` 管理 UI 状态、`SharedFlow(replay=0)` 处理一次性事件，在 Fragment 中用 `repeatOnLifecycle(STARTED)` 安全收集。LiveData 的粘性事件问题（新 Observer 收到旧数据）在 SharedFlow 中不存在。

## 二、Android 篇

### Q5：说说 Activity 的四种启动模式？举例说明使用场景

**🎯 关键词**：任务栈复用、onNewIntent、taskAffinity

> **回答**：
>
> - **standard**：默认模式，每次 startActivity 创建新实例。大多数普通页面用这个。
> - **singleTop**：栈顶复用。如果目标 Activity 已在栈顶则调用 `onNewIntent()` 而不是新建。适合**搜索结果页**——连续搜索不应创建多个实例、**推送通知点击页**。
> - **singleTask**：栈内复用。栈中已有则清除其上所有 Activity 并调用 `onNewIntent()`。适合**首页/主页面**——从任何深层页面回到首页时清栈。
> - **singleInstance**：独占一个任务栈。适合**来电界面、系统级入口**等全局唯一的页面。
>
> 补充：可以通过 Intent Flag 动态控制，如 `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_NEW_TASK` 约等于 singleTask 效果。面试中如果问到 `taskAffinity`，它决定 Activity 归属哪个任务栈，需配合 `singleTask` 或 `FLAG_ACTIVITY_NEW_TASK` 才生效。

### Q6：Handler 消息机制中，为什么 Looper.loop() 死循环不会 ANR？

**🎯 关键词**：epoll 阻塞、CPU 不空转、ANR 是消息处理超时

> **回答**：这是一个常见的混淆。ANR 是指**某个消息处理时间过长**（如 5 秒内未响应输入事件），而不是"线程在循环"。Looper.loop() 的死循环在没有消息时会调用 `MessageQueue.next()`，底层通过 Linux 的 `epoll_wait()` 系统调用**阻塞等待**，此时线程处于 SLEEPING 状态，CPU 占用率为 0。当有新消息入队（如触摸事件、生命周期回调、View 绘制），通过 `nativeWake()` 唤醒 epoll，线程醒来处理消息。所以主线程的一切——包括 Activity 生命周期、触摸事件、UI 绘制——本质上都是一个个 Message 在 Handler 消息循环中被处理。真正导致 ANR 的是某个 `handleMessage()` 执行耗时操作。

### Q7：RecyclerView 和 ListView 的区别？RecyclerView 的缓存机制？

**🎯 关键词**：四级缓存、ViewHolder 强制、DiffUtil

> **回答**：
> **区别**：ListView 不强制使用 ViewHolder；RecyclerView 强制分离了 LayoutManager（布局）、Adapter（数据绑定）、ItemAnimator（动画）、ItemDecoration（装饰），更灵活。RecyclerView 支持线性、网格、瀑布流布局。
>
> **四级缓存**：
>
> 1. **Scrap**（屏幕内可见的 ViewHolder）——刷新时暂存，不需要重新绑定
> 2. **CachedViews**（默认 2 个）——按 position 匹配，命中后**不需要 bind**，直接复用
> 3. **ViewCacheExtension**——自定义缓存，很少用
> 4. **RecycledViewPool**（每种 viewType 默认 5 个）——按 viewType 匹配，命中后**需要重新 bind**
>
> 都没命中则走 `createViewHolder()` + `bindViewHolder()`，开销最大。性能优化的核心是增大缓存命中率、减少 bind 和 create 的执行次数，以及使用 DiffUtil 精确更新避免 `notifyDataSetChanged()`。

### Q8：事件分发中如何解决滑动冲突？

**🎯 关键词**：外部拦截法、内部拦截法、onInterceptTouchEvent

> **回答**：有两种标准方案：
>
> **外部拦截法**（推荐）：重写父 View 的 `onInterceptTouchEvent`。DOWN 事件不拦截（否则子 View 收不到），在 MOVE 事件中根据手势方向或距离判断是否拦截。例如 ViewPager 嵌套 RecyclerView 时，水平滑动距离 > 垂直距离则拦截给 ViewPager 处理。
>
> **内部拦截法**：子 View 在 `dispatchTouchEvent` 中拦。DOWN 时调用 `parent.requestDisallowInterceptTouchEvent(true)` 禁止父 View 拦截；在 MOVE 中判断不需要自己处理时放开拦截 `(false)`。需要父 View 配合在仅在 DOWN 时不拦截。
>
> 实际开发中外部拦截法更直观，也是 Google 推荐的方式。

### Q9：Jetpack ViewModel 是如何在配置变更后存活的？

**🎯 关键词**：NonConfigurationInstances、ViewModelStore、onRetainNonConfigurationInstance

> **回答**：Activity 因配置变更销毁时，系统会调用 `onRetainNonConfigurationInstance()` 保存 `ViewModelStore` 对象到一个名为 `NonConfigurationInstances` 的容器中。这个容器通过 `ActivityClientRecord` 保持引用（在 ActivityThread 中，进程级别保活）。新 Activity 创建时，通过 `getLastNonConfigurationInstance()` 取回同一个 `ViewModelStore`，于是里面的 ViewModel 实例不变。但如果是用户主动 finish 或系统杀进程，ViewModelStore 会被清除，ViewModel 销毁。需要跨进程死亡保存状态的话要结合 `SavedStateHandle`。

## 三、架构与综合篇

### Q10：你在项目中使用什么架构？为什么？

**🎯 关键词**：单向数据流、可测试性、关注点分离

> **回答模板**（根据你的实际经验调整）：
>
> "我们项目采用 **MVVM + Clean Architecture** 分层。
>
> **Presentation 层**：Activity/Fragment 只负责 UI 渲染和事件转发，ViewModel 通过 StateFlow 暴露不可变的 UiState；
>
> **Domain 层**：纯 Kotlin 模块，定义 UseCase 和 Repository 接口，不依赖 Android Framework，方便单元测试；
>
> **Data 层**：实现 Repository 接口，协调 RemoteDataSource（Retrofit）和 LocalDataSource（Room），实现缓存策略。
>
> 选择这个架构的原因：
>
> 1. **单向数据流**使状态可预测，Debug 方便
> 2. **Domain 层独立**可以 100% 纯 JVM 单元测试
> 3. **依赖反转**使得替换数据源（如从 REST 换到 GraphQL）不影响上层
> 4. 配合 Hilt 依赖注入，各层之间松耦合
>
> 如果项目更复杂（如多数据源实时同步），我会考虑 MVI 模式让状态管理更集中。"

## 四、系统思维题

### Q11：从输入 URL 到页面显示，经历了什么？（Android WebView 版本）

> 1. **DNS 解析**：域名 → IP 地址（本地缓存 → 系统缓存 → DNS 服务器递归查询）
> 2. **TCP 连接**：三次握手建立连接
> 3. **TLS 握手**：交换密钥，建立加密通道（如果是 HTTPS）
> 4. **HTTP 请求**：发送 GET 请求，服务器返回 HTML
> 5. **HTML 解析**：构建 DOM 树
> 6. **CSS 解析**：构建 CSSOM 树
> 7. **渲染树合成**：DOM + CSSOM → Render Tree
> 8. **布局（Layout/Reflow）**：计算每个节点的位置和大小
> 9. **绘制（Paint）**：渲染到屏幕
> 10. **JavaScript 执行**：可能修改 DOM 触发回流重绘

### Q12：App 启动流程？

> 1. 用户点击图标 → **Launcher** 发起 startActivity
> 2. **AMS**（ActivityManagerService）处理启动请求
> 3. **Zygote** fork 出新进程
> 4. 新进程中初始化 **ActivityThread**，创建主线程 Looper
> 5. 通过反射创建 **Application** 对象，调用 `onCreate()`
> 6. AMS 通知 App 启动目标 **Activity**
> 7. 创建 Activity 对象，调用 `onCreate()` → `onStart()` → `onResume()`
> 8. ViewRootImpl 执行首次 **measure → layout → draw**
> 9. 首帧渲染完成 → 用户看到界面

### Q13：如何优化 App 启动速度？

> 1. **Application.onCreate()**：异步初始化非关键 SDK（使用协程或 App Startup）
> 2. **减少主线程阻塞**：避免同步 I/O、大量计算
> 3. **Splash Screen**：使用 SplashScreen API 而非沉重的 Activity
> 4. **延迟初始化**：通过 `IdleHandler` 在空闲时初始化
> 5. **减少布局层级**：首页 Layout 使用 ConstraintLayout + ViewStub
> 6. **类加载优化**：减少 MultiDex 影响
> 7. **资源优化**：压缩图片、移除无用资源
