# Kotlin 协程（Coroutines）

## 一、协程基础概念

### 1. 什么是协程？

> **面试标准回答**：协程是一种**轻量级的线程替代方案**，本质是**编译器通过 CPS（Continuation Passing Style）变换**实现的可挂起计算。它可以在不阻塞线程的情况下暂停和恢复执行。

**协程 vs 线程**：

| 对比 | 线程 | 协程 |
|------|------|------|
| 调度 | OS 内核调度 | 用户态调度（协程调度器） |
| 内存开销 | ~1MB 栈空间 | ~几十字节对象 |
| 切换代价 | 内核态/用户态切换 | 函数调用级别 |
| 创建数量 | 数千级别 | 数十万级别 |
| 阻塞 | 线程阻塞 | 挂起不阻塞线程 |

### 2. 协程构建器

```kotlin
// launch：启动协程，返回 Job，不返回结果（fire-and-forget）
val job: Job = scope.launch {
    delay(1000)
    println("World")
}

// async：启动协程，返回 Deferred<T>，可获取结果
val deferred: Deferred<String> = scope.async {
    delay(1000)
    "Hello"
}
val result = deferred.await()

// runBlocking：阻塞当前线程（主要用于测试和 main 函数）
fun main() = runBlocking {
    launch { delay(1000); println("World") }
    println("Hello")
}

// withContext：切换上下文并返回结果（不创建新协程）
suspend fun fetchData(): String = withContext(Dispatchers.IO) {
    // 在 IO 线程执行
    api.getData()
}
```

## 二、挂起函数（suspend function）

> **面试核心考点**：suspend 关键字的本质是什么？

### 1. suspend 的本质

```kotlin
// 声明
suspend fun fetchUser(): User {
    delay(1000)  // 挂起点
    return User("John")
}

// 编译后等价伪代码（CPS 变换）
fun fetchUser(continuation: Continuation<User>): Any? {
    // 状态机实现
    when (continuation.label) {
        0 -> {
            continuation.label = 1
            if (delay(1000, continuation) == COROUTINE_SUSPENDED)
                return COROUTINE_SUSPENDED
        }
        1 -> {
            return User("John")
        }
    }
}
```

**关键点**：

1. `suspend` 函数编译后多了一个 `Continuation` 参数
2. 函数体被编译为**状态机**，每个挂起点是一个状态
3. 挂起时返回 `COROUTINE_SUSPENDED` 标记
4. 恢复时从上次的状态继续执行

### 2. Continuation 接口

```kotlin
public interface Continuation<in T> {
    public val context: CoroutineContext
    public fun resumeWith(result: Result<T>)
}

// 扩展函数
fun <T> Continuation<T>.resume(value: T)
fun <T> Continuation<T>.resumeWithException(exception: Throwable)
```

## 三、协程调度器（Dispatcher）

> **面试常问**：各种 Dispatcher 的区别和使用场景？

```kotlin
Dispatchers.Main        // 主线程（UI 操作），Android 中由 Handler 实现
Dispatchers.IO          // IO 密集型，线程池上限 64（或 CPU 核心数的较大值）
Dispatchers.Default     // CPU 密集型，线程数 = CPU 核心数
Dispatchers.Unconfined  // 不切换线程，在调用线程执行（测试用，不推荐生产使用）
```

| Dispatcher | 线程池大小 | 适用场景 |
|-----------|----------|---------|
| `Main` | 主线程 | UI 更新、LiveData、Toast |
| `IO` | 最大 64 | 网络请求、文件读写、数据库操作 |
| `Default` | CPU 核心数 | 大量计算、JSON 解析、列表排序 |
| `Unconfined` | 无固定线程 | 测试、不需要特定线程的计算 |

```kotlin
// IO 和 Default 共享线程池，切换时不一定产生线程切换
suspend fun processData() {
    val data = withContext(Dispatchers.IO) {
        api.fetchData()  // IO 线程
    }
    withContext(Dispatchers.Default) {
        data.sort()      // CPU 密集操作
    }
    withContext(Dispatchers.Main) {
        updateUI(data)   // 主线程更新 UI
    }
}
```

## 四、结构化并发

### 1. CoroutineScope

> **面试常问**：什么是结构化并发？为什么重要？

```kotlin
// 结构化并发确保：
// 1. 父协程取消时，所有子协程自动取消
// 2. 子协程出错时，父协程和兄弟协程也被取消
// 3. 父协程等待所有子协程完成后才完成

// ❌ 错误做法：使用 GlobalScope（非结构化）
GlobalScope.launch {
    // 生命周期跟随进程，会导致内存泄漏
}

// ✅ 正确做法：使用适当的 Scope
class MyViewModel : ViewModel() {
    fun loadData() {
        viewModelScope.launch {  // ViewModel 销毁时自动取消
            val user = fetchUser()
            _userLiveData.value = user
        }
    }
}

class MyFragment : Fragment() {
    fun loadData() {
        viewLifecycleOwner.lifecycleScope.launch {
            // Fragment View 销毁时自动取消
        }
    }
}
```

### 2. Job 层级

```kotlin
val parentJob = scope.launch {
    val child1 = launch {
        delay(1000)
        println("Child 1 完成")
    }
    val child2 = launch {
        delay(2000)
        println("Child 2 完成")
    }
    // parentJob 等待 child1 和 child2 都完成
}

// 取消传播
parentJob.cancel()  // child1 和 child2 也被取消
```

### 3. SupervisorJob

> **面试常问**：Job 和 SupervisorJob 的区别？

```kotlin
// Job：一个子协程失败 → 取消所有兄弟 + 父协程
// SupervisorJob：一个子协程失败 → 不影响其他子协程

val supervisor = SupervisorJob()
val scope = CoroutineScope(Dispatchers.Main + supervisor)

scope.launch {
    throw Exception("Child 1 失败")  // 不影响 child2
}
scope.launch {
    delay(1000)
    println("Child 2 正常运行")  // 仍然执行
}

// supervisorScope 函数
suspend fun loadMultipleData() = supervisorScope {
    val users = async { fetchUsers() }      // 失败不影响 posts
    val posts = async { fetchPosts() }      // 失败不影响 users
    try {
        updateUI(users.await(), posts.await())
    } catch (e: Exception) {
        // 处理部分失败
    }
}
```

## 五、协程异常处理

> **面试高频考点**：协程中异常如何传播和处理？

### 1. 异常传播规则

```kotlin
// launch：异常自动传播到父协程
// async：异常在 await() 时抛出

// CoroutineExceptionHandler（仅对 launch 有效，且仅在根协程上设置）
val handler = CoroutineExceptionHandler { _, throwable ->
    Log.e("Coroutine", "捕获异常: ${throwable.message}")
}

val scope = CoroutineScope(Dispatchers.Main + handler)
scope.launch {
    throw RuntimeException("Boom!")  // 被 handler 捕获
}
```

### 2. try-catch 的正确使用

```kotlin
// ✅ 在协程内部 try-catch
scope.launch {
    try {
        val result = riskyOperation()
    } catch (e: Exception) {
        // 处理异常
    }
}

// ✅ async + try-catch await
scope.launch {
    val deferred = async { riskyOperation() }
    try {
        val result = deferred.await()
    } catch (e: Exception) {
        // 处理异常
    }
}

// ❌ 在外部 try-catch launch（无效！）
try {
    scope.launch {
        throw RuntimeException("Boom!")
    }
} catch (e: Exception) {
    // 捕获不到！launch 内的异常不会传播到这里
}
```

## 六、Flow（冷流）

> **面试重点**：Flow vs LiveData vs Channel

### 1. Flow 基础

```kotlin
// 创建 Flow
fun fetchNumbers(): Flow<Int> = flow {
    for (i in 1..5) {
        delay(100)
        emit(i)  // 发射值
    }
}

// 收集 Flow
scope.launch {
    fetchNumbers().collect { value ->
        println(value)
    }
}

// flowOf / asFlow
val simpleFlow = flowOf(1, 2, 3)
val listFlow = listOf(1, 2, 3).asFlow()
```

### 2. Flow 操作符

```kotlin
fetchNumbers()
    .map { it * 2 }                    // 变换
    .filter { it > 4 }                 // 过滤
    .onEach { log("Received: $it") }   // 副作用
    .catch { e -> emit(-1) }           // 异常处理
    .onStart { emit(0) }              // 开始时
    .onCompletion { cause -> }         // 完成时
    .flowOn(Dispatchers.IO)            // 上游切换线程
    .collect { value -> updateUI(value) }

// debounce（搜索输入防抖）
searchEditText.textChanges()
    .debounce(300)
    .distinctUntilChanged()
    .flatMapLatest { query ->
        searchApi(query)
    }
    .collect { results ->
        showResults(results)
    }
```

### 3. StateFlow 与 SharedFlow

> **面试常问**：StateFlow vs LiveData？

```kotlin
// StateFlow：有状态的热流（替代 LiveData）
class MyViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(UiState.Loading)
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun loadData() {
        viewModelScope.launch {
            _uiState.value = UiState.Loading
            try {
                val data = repository.fetchData()
                _uiState.value = UiState.Success(data)
            } catch (e: Exception) {
                _uiState.value = UiState.Error(e.message)
            }
        }
    }
}

// Fragment 中收集
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collect { state ->
            when (state) {
                is UiState.Loading -> showLoading()
                is UiState.Success -> showData(state.data)
                is UiState.Error -> showError(state.message)
            }
        }
    }
}
```

| 对比 | `LiveData` | `StateFlow` | `SharedFlow` |
|------|-----------|-------------|-------------|
| 平台 | Android 特有 | Kotlin 多平台 | Kotlin 多平台 |
| 初始值 | 可选 | 必须 | 无 |
| 去重 | 无 | `distinctUntilChanged` | 可配置 |
| 生命周期感知 | 自动 | 需配合 `repeatOnLifecycle` | 需配合 |
| 回放 | 最新值 | 最新值 | 可配置 `replay` |
| 适用场景 | UI 状态 | UI 状态（多平台） | 事件（一次性消息） |

### 3. 重试机制

```kotlin
suspend fun <T> retry(
    times: Int = 3,
    initialDelay: Long = 100,
    maxDelay: Long = 1000,
    factor: Double = 2.0,
    block: suspend () -> T
): T {
    var currentDelay = initialDelay
    repeat(times - 1) {
        try {
            return block()
        } catch (e: Exception) {
            // 日志记录
        }
        delay(currentDelay)
        currentDelay = (currentDelay * factor).toLong().coerceAtMost(maxDelay)
    }
    return block()  // 最后一次抛出异常
}
```

### 4. 取消与清理

```kotlin
suspend fun downloadFile(url: String) {
    val connection = openConnection(url)
    try {
        // ensureActive() 检查取消状态
        ensureActive()
        while (isActive) {  // 协程是否仍活跃
            val chunk = connection.readChunk()
            if (chunk == null) break
            writeToFile(chunk)
            yield()  // 给取消一个机会
        }
    } finally {
        // 无论是否取消都执行清理
        withContext(NonCancellable) {
            connection.close()  // 即使被取消也执行
        }
    }
}
```
