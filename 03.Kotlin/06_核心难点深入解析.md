## 一、协程挂起原理（彻底搞懂 suspend）

### 🎯 一句话总结
>
> `suspend` 函数 = 编译器把你的代码拆成**状态机**，在挂起点暂停，把"接下来要做什么"打包成一个 `Continuation` 对象传递。

### 📖 生活类比

想象你在餐厅点餐：

1. 你对服务员说"我要一份牛排"（发起异步操作）
2. 服务员给你一个**取餐号**（Continuation），你不需要站在窗口等
3. 你回到座位继续聊天（线程去做别的事）
4. 牛排做好了，服务员通过取餐号找到你（resume），把牛排送来
5. 你继续吃饭（从挂起点恢复执行）

**关键**：你（线程）在等待期间**没有被阻塞**，可以做其他事。

### 🔬 逐步拆解编译过程

#### 原始代码

```kotlin
suspend fun loadUserData(): String {
    val token = getToken()           // 挂起点1
    val user = getUser(token)        // 挂起点2
    return "Hello, ${user.name}"
}
```

#### 编译后的状态机（伪代码）

```kotlin
// 编译器自动生成，你不需要手写！
fun loadUserData(continuation: Continuation<String>): Any? {
    // 编译器创建的状态机类
    class LoadUserDataStateMachine(
        completion: Continuation<String>
    ) : ContinuationImpl(completion) {
        var label = 0          // 当前状态（执行到第几步）
        var token: String? = null   // 中间结果暂存
        var user: User? = null
    }

    // 首次调用 or 恢复执行
    val sm = continuation as? LoadUserDataStateMachine
        ?: LoadUserDataStateMachine(continuation)

    when (sm.label) {
        0 -> {
            // ===== 第一段代码：调用 getToken() =====
            sm.label = 1                           // 设置下一个状态
            val result = getToken(sm)              // 传入自己作为回调
            if (result == COROUTINE_SUSPENDED) {
                return COROUTINE_SUSPENDED         // 真的挂起了，函数返回
                // ⚡ 此时线程被释放，去做别的事了！
            }
            // 没有真正挂起（同步返回），继续往下
            sm.token = result as String
        }
        1 -> {
            // ===== getToken() 完成，恢复到这里 =====
            sm.token = sm.result as String         // 取出 getToken 的结果
            sm.label = 2
            val result = getUser(sm.token!!, sm)   // 调用 getUser
            if (result == COROUTINE_SUSPENDED) {
                return COROUTINE_SUSPENDED
            }
            sm.user = result as User
        }
        2 -> {
            // ===== getUser() 完成，恢复到这里 =====
            sm.user = sm.result as User
            return "Hello, ${sm.user!!.name}"     // 最终结果
        }
    }
}
```

#### 执行流程图

```
线程A                          线程B (IO)
  │
  │ loadUserData(continuation)
  │     label=0
  │     调用 getToken(sm)
  │     ← COROUTINE_SUSPENDED
  │     ⚡ 线程A被释放做其他事
  │                              │
  │                              │ getToken 网络请求完成
  │                              │ sm.resumeWith("abc123")
  │                              │
  │ ← 恢复执行（可能是线程A或C）
  │     label=1
  │     token = "abc123"
  │     调用 getUser("abc123", sm)
  │     ← COROUTINE_SUSPENDED
  │     ⚡ 线程再次被释放
  │                              │
  │                              │ getUser 完成
  │                              │ sm.resumeWith(User("张三"))
  │
  │ ← 恢复执行
  │     label=2
  │     return "Hello, 张三"
  │
  ✅ 完成
```

### 💡 面试回答模板

> "suspend 函数本质上是 **CPS 变换 + 状态机**。编译器在每个挂起点把函数拆分成多个状态（label），函数多一个 `Continuation` 参数。挂起时返回 `COROUTINE_SUSPENDED` 标记释放线程；异步操作完成后通过 `Continuation.resumeWith()` 恢复到下一个状态继续执行。这就是为什么协程不阻塞线程——线程只是把后续任务打包好放一旁，空闲去做其他事。"

## 二、泛型型变（in/out）彻底搞懂

### 🎯 一句话总结
>
> - `out T` = 只能**生产/输出** T → 协变 → 子类型安全（`List<Dog>` 可当 `List<Animal>` 用）
> - `in T` = 只能**消费/输入** T → 逆变 → 父类型安全（`Comparator<Animal>` 可当 `Comparator<Dog>` 用）

### 📖 生活类比 — 水果篮子

#### 协变 out（只取不放 → 安全放宽）

```
🍎 苹果篮子 (List<Apple>)
  → 可以当成"水果篮子"来用 (List<Fruit>)
  → 因为你只是 **取出** 东西，取出来的苹果一定是水果 ✅

  → 但不能 **放入** 东西！
  → 如果允许放入，你可能往"苹果篮子"里放了橘子 ❌
```

```kotlin
// List 声明为 out T（只读，只取不放）
interface List<out T> {
    fun get(index: Int): T          // ✅ T 在返回值位置
    // fun add(element: T)          // ❌ 编译错误！out 不能做参数
}

val apples: List<Apple> = listOf(Apple(), Apple())
val fruits: List<Fruit> = apples   // ✅ 协变！安全
val thing: Fruit = fruits[0]       // ✅ 取出的苹果一定是水果
```

#### 逆变 in（只放不取 → 安全收窄）

```
🗑️ 水果垃圾桶 (Consumer<Fruit>)
  → 可以当成"苹果垃圾桶"来用 (Consumer<Apple>)
  → 因为能吃水果的嘴，一定也能吃苹果 ✅

  → 但不能 **取出** 东西！
  → 取出来你不知道是苹果还是橘子 ❌
```

```kotlin
// Comparable 声明为 in T（只消费，只放不取）
interface Comparable<in T> {
    fun compareTo(other: T): Int    // ✅ T 在参数位置
    // fun getItem(): T             // ❌ 编译错误！in 不能做返回值
}

val fruitComparator: Comparable<Fruit> = /* 按重量比较水果 */
val appleComparator: Comparable<Apple> = fruitComparator  // ✅ 逆变！
// 能比较水果的，一定能比较苹果
```

### 🔑 核心记忆口诀

```
┌─────────────────────────────────────────────┐
│                                             │
│   Producer → out → 输出 → 协变 → 子可赋父   │
│   Consumer → in  → 输入 → 逆变 → 父可赋子   │
│                                             │
│   Java 对应：                               │
│   out ↔ ? extends T                         │
│   in  ↔ ? super T                           │
│                                             │
│   记忆：PECS                                │
│   Producer Extends, Consumer Super          │
│                                             │
└─────────────────────────────────────────────┘
```

### 📊 完整对比表

```kotlin
// 不变（Invariant）：MutableList<Dog> ≠ MutableList<Animal>
// 因为既能放又能取，放宽任何方向都不安全
val dogs: MutableList<Dog> = mutableListOf()
// val animals: MutableList<Animal> = dogs  // ❌ 编译错误

// 协变（Covariant）：List<Dog> 是 List<Animal> 的子类型
val readOnlyDogs: List<Dog> = listOf()
val readOnlyAnimals: List<Animal> = readOnlyDogs  // ✅

// 逆变（Contravariant）：Comparable<Animal> 是 Comparable<Dog> 的子类型
val animalComparator: Comparable<Animal> = compareBy { it.name }
val dogComparator: Comparable<Dog> = animalComparator  // ✅
```

### 💡 面试回答模板

> "`out` 代表协变，只能在输出位置使用（返回值），使得 `List<子类>` 可以赋给 `List<父类>`，因为你只取不放，取出的子类一定是父类的实例。`in` 代表逆变，只能在输入位置使用（参数），使得 `Comparable<父类>` 可以赋给 `Comparable<子类>`，因为能处理父类的，一定能处理子类。不声明 in/out 就是不变，既读又写时不安全。"

## 三、inline + reified 的本质

### 🎯 一句话总结
>
> `inline` = 把函数体直接**复制粘贴**到调用处，省去 Lambda 对象的创建开销。
> `reified` = 因为代码被粘贴了，泛型类型在粘贴时可以**被替换为具体类型**，所以能在运行时使用。

### 📖 生活类比

**没有 inline**：你在网上下单买东西（创建 Lambda 对象），快递送到（间接调用），拆包（执行），这中间有很多开销。

**有了 inline**：你直接走到架子旁边拿（代码直接嵌入），没有包装、没有快递、没有拆包。

### 🔬 编译前后对比

```kotlin
// 定义
inline fun <reified T> isType(value: Any): Boolean = value is T

// 调用
val result = isType<String>("hello")

// ----- 编译后等价于（代码直接展开）-----
val result = "hello" is String    // 泛型 T 被替换为 String！
```

```kotlin
// 没有 inline 时的 Lambda 开销
fun doSomething(action: () -> Unit) {   // 非内联
    action()
}
doSomething { println("hello") }

// 编译后：
doSomething(new Function0<Unit>() {     // 创建匿名类对象！
    @Override
    public Unit invoke() {
        System.out.println("hello");    // 间接调用
        return Unit.INSTANCE;
    }
});

// 有 inline 时
inline fun doSomethingInline(action: () -> Unit) {
    action()
}
doSomethingInline { println("hello") }

// 编译后：直接展开，没有创建对象！
System.out.println("hello");
```

### 💡 什么时候该用 inline？

| 场景 | 是否 inline | 原因 |
|------|------------|------|
| 参数有 Lambda | ✅ 推荐 | 消除 Lambda 对象开销 |
| 需要 `reified` | ✅ 必须 | reified 只能用在 inline 函数中 |
| 函数体很大 | ❌ 不推荐 | 代码膨胀（每处调用都复制） |
| 没有 Lambda 参数 | ❌ 不推荐 | 没有优化价值 |

## 四、协程 Flow 的冷流 vs 热流

### 📖 生活类比

**冷流（Flow）** = 点播电影 🎬

- 你什么时候点播，什么时候开始放
- 每个观众都从头看（独立的数据流）
- 没人看就不制作

**热流（StateFlow/SharedFlow）** = 直播频道 📺

- 不管有没有人看，直播一直在进行
- 新观众从当前画面开始看（错过之前的）
- `StateFlow` = 永远显示最新一帧的电视
- `SharedFlow` = 可配置"回放最近几帧"的直播

```kotlin
// 冷流：每次 collect 都重新执行
val coldFlow = flow {
    println("开始制作数据")  // 每次 collect 都会执行
    emit(1)
    emit(2)
}
coldFlow.collect { }  // 打印"开始制作数据"
coldFlow.collect { }  // 再次打印"开始制作数据"

// 热流：数据独立于收集者存在
val hotFlow = MutableStateFlow(0)
hotFlow.value = 1     // 即使没有收集者也可以更新
hotFlow.value = 2
// 新的收集者只能看到最新值 2
hotFlow.collect { }   // 收到 2
```

### 💡 选择指南

```
需要 UI 状态？ → StateFlow（总有一个最新值）
需要一次性事件（Toast、导航）？ → SharedFlow(replay=0)
需要数据库/网络请求的响应式数据？ → Flow（冷流）
和 Java 代码交互？ → LiveData（或用 asLiveData() 转换）
```

## 五、结构化并发 — 为什么 GlobalScope 是错的？

### 📖 生活类比

**GlobalScope** = 你雇了一个人去干活，但你走了他还在干，你不知道他做完了没有，出了问题你也管不了。这个人如果在你的餐厅里操作，你关了店他还在里面做事——**危险！**

**viewModelScope** = 你雇了一个人坐在你的办公室里干活。你下班（ViewModel 销毁）的时候，他自动离开。出了问题你可以随时叫停。

### 🔬 实际问题演示

```kotlin
// ❌ GlobalScope 的问题
class MyFragment : Fragment() {
    fun loadData() {
        GlobalScope.launch {
            val data = api.fetchData()  // 网络请求
            // ⚠️ Fragment 已经销毁了，但这个协程还在跑！
            binding.textView.text = data  // 💥 崩溃！binding 已为 null
        }
    }
}

// ✅ 正确做法
class MyFragment : Fragment() {
    fun loadData() {
        viewLifecycleOwner.lifecycleScope.launch {
            val data = api.fetchData()
            // Fragment view 销毁时自动取消，不会执行到这里
            binding.textView.text = data  // ✅ 安全
        }
    }
}
```

### 📊 作用域选择表

| 作用域 | 生命周期跟随 | 使用场景 |
|--------|------------|---------|
| `viewModelScope` | ViewModel | 数据加载、业务逻辑 |
| `lifecycleScope` | Activity/Fragment | UI 相关操作 |
| `viewLifecycleOwner.lifecycleScope` | Fragment 的 View | 最安全的 Fragment UI 操作 |
| `GlobalScope` | 应用进程 | ⚠️ 几乎不应该使用 |
| `CoroutineScope(Job())` | 自行管理 | 自定义生命周期组件 |

## 六、协程异常处理 — 最容易搞混的地方

### 📖 核心规则图

```
                    launch             async
                      │                  │
                 自动传播异常          await() 时抛出
                      │                  │
              CoroutineExceptionHandler   try-catch await()
                      │
              ⚠️ 只在根协程上有效！

    异常传播方向：
    子协程 ──异常──→ 父协程 ──异常──→ 爷协程
         ← 取消 ←        ← 取消 ←
    （一个子出错 → 取消所有兄弟 → 取消父亲）

    SupervisorJob：
    子协程 ──异常──→ ❌ 不传给父级
    其他子协程正常运行 ✅
```

### 🔬 常见错误

```kotlin
// ❌ 错误1：在内层 launch 设置 handler（无效！）
scope.launch {
    launch(handler) {  // ❌ handler 在非根协程上，无效！
        throw Exception()
    }
}

// ✅ 正确：handler 设在根协程上
scope.launch(handler) {
    launch {
        throw Exception()  // ✅ 被根协程的 handler 捕获
    }
}

// ❌ 错误2：try-catch 包裹 launch（无效！）
try {
    scope.launch {
        throw Exception()  // launch 内的异常不会被外面的 try 捕获
    }
} catch (e: Exception) {
    // 捕获不到！❌
}

// ✅ 正确：try-catch 放在 launch 里面
scope.launch {
    try {
        riskyWork()
    } catch (e: Exception) {
        // ✅ 可以捕获
    }
}
```
