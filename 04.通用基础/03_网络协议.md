# 网络协议面试要点

## 一、TCP 与 UDP

### 1. TCP 三次握手

```
Client                  Server
  │── SYN(seq=x) ──→      │   第一次：客户端请求建立连接
  │                        │
  │←── SYN+ACK ────       │   第二次：服务端确认 + 请求建立
  │    (seq=y, ack=x+1)    │
  │                        │
  │── ACK(ack=y+1) ──→    │   第三次：客户端确认
  │                        │
  │←── 数据传输 ────→     │
```

> **面试常问**：为什么是三次而不是两次？
>
> - 防止**已失效的请求报文**到达服务器导致错误建连
> - 两次握手无法确认客户端的接收能力

### 2. TCP 四次挥手

```
Client                  Server
  │── FIN ──────→          │   第一次：客户端请求关闭
  │←── ACK ─────          │   第二次：服务端确认
  │                        │   （服务端可能还有数据要发送）
  │←── FIN ─────          │   第三次：服务端请求关闭
  │── ACK ──────→          │   第四次：客户端确认
  │ TIME_WAIT(2MSL)        │
```

> **面试常问**：为什么需要 TIME_WAIT？
>
> - 确保最后一个 ACK 到达服务端（丢失时服务端会重发 FIN）
> - 等待 2MSL 确保旧连接的所有报文在网络中消失

### 3. TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（重传、排序） | 不可靠 |
| 顺序 | 保证有序 | 不保证 |
| 速度 | 相对慢 | 快 |
| 头部开销 | 20 字节 | 8 字节 |
| 适用场景 | HTTP、文件传输 | 视频直播、DNS、游戏 |

## 二、HTTP 协议

### 1. HTTP 请求方法

| 方法 | 幂等 | 安全 | 用途 |
|------|------|------|------|
| GET | ✅ | ✅ | 获取资源 |
| POST | ❌ | ❌ | 创建资源 |
| PUT | ✅ | ❌ | 更新资源（全量） |
| PATCH | ❌ | ❌ | 更新资源（部分） |
| DELETE | ✅ | ❌ | 删除资源 |
| HEAD | ✅ | ✅ | 获取响应头 |
| OPTIONS | ✅ | ✅ | 跨域预检请求 |

### 2. HTTP 状态码

| 分类 | 含义 | 常见 |
|------|------|------|
| 1xx | 信息 | 100 Continue |
| 2xx | 成功 | 200 OK, 201 Created, 204 No Content |
| 3xx | 重定向 | 301 永久, 302 临时, 304 缓存未修改 |
| 4xx | 客户端错误 | 400 Bad Request, 401 未认证, 403 禁止, 404 Not Found |
| 5xx | 服务端错误 | 500 Internal, 502 Bad Gateway, 503 Service Unavailable |

### 3. HTTP 缓存机制

```
强缓存（不发请求）：
Cache-Control: max-age=3600        // 优先级高
Expires: Thu, 01 Dec 2025 16:00:00 GMT  // 绝对时间

协商缓存（发请求，可能 304）：
请求 → If-None-Match: "etag-value"
响应 → ETag: "etag-value"

请求 → If-Modified-Since: Thu, 01 Dec 2025
响应 → Last-Modified: Thu, 01 Dec 2025

判断流程：
有缓存 → 强缓存未过期？→ 直接使用（200 from cache）
        → 过期 → 协商缓存 → 未变化？→ 304 Not Modified
                            → 已变化 → 200 + 新数据
```

### 4. HTTP/1.1 vs HTTP/2 vs HTTP/3

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|---------|--------|--------|
| 传输层 | TCP | TCP | **QUIC (UDP)** |
| 多路复用 | ❌（队头阻塞） | ✅ | ✅ |
| 头部压缩 | ❌ | HPACK | QPACK |
| 服务器推送 | ❌ | ✅ | ✅ |
| 连接建立 | TCP + TLS | TCP + TLS | 0-RTT / 1-RTT |

## 三、HTTPS

### 1. HTTPS = HTTP + TLS

```
TLS 握手简化流程：

1. Client → Server：支持的TLS版本、加密套件、随机数A
2. Server → Client：选择的版本/套件、随机数B、服务器证书
3. Client 验证证书 → 生成预主密钥 → 用证书公钥加密发送
4. 双方用 随机数A + 随机数B + 预主密钥 → 生成会话密钥
5. 后续通信使用对称加密（AES）
```

### 2. 对称加密 vs 非对称加密

| 对比 | 对称加密 | 非对称加密 |
|------|---------|----------|
| 密钥 | 共享一把密钥 | 公钥+私钥 |
| 速度 | 快 | 慢（约100倍差距） |
| 典型算法 | AES、DES | RSA、ECC |
| HTTPS中 | 加密数据传输 | 交换对称密钥 |

### 3. 证书验证流程

1. 服务器发送 CA 签发的证书
2. 客户端用 CA 公钥解密签名，获取证书哈希
3. 客户端计算证书内容的哈希
4. 两个哈希比对 → 一致则证书合法

## 四、WebSocket

```kotlin
// 全双工通信，适用于实时场景（聊天、推送）
// 基于 HTTP 升级握手

// OkHttp WebSocket
val client = OkHttpClient()
val request = Request.Builder().url("ws://example.com/ws").build()
val ws = client.newWebSocket(request, object : WebSocketListener() {
    override fun onOpen(webSocket: WebSocket, response: Response) {
        webSocket.send("Hello Server")
    }
    override fun onMessage(webSocket: WebSocket, text: String) {
        println("收到: $text")
    }
    override fun onFailure(webSocket: WebSocket, t: Throwable, response: Response?) {
        println("连接失败: ${t.message}")
    }
})
```

## 五、DNS 解析

```
用户输入 URL →
  1. 浏览器/App 缓存
  2. 操作系统缓存（hosts 文件）
  3. 本地 DNS 服务器
  4. 根 DNS → 顶级域 DNS → 权威 DNS
  ← 返回 IP 地址
```

**Android 中的 DNS 优化**：

- 使用 `HttpDns`（避免运营商 DNS 劫持）
- 预解析（App 启动时提前解析核心域名）
- IP 直连（跳过 DNS，但注意 HTTPS 证书校验）

## 六、Cookie、Session、Token

| 对比 | Cookie | Session | Token (JWT) |
|------|--------|---------|-------------|
| 存储 | 客户端 | 服务端 | 客户端 |
| 安全性 | 较低 | 较高 | 较高（签名） |
| 跨域 | 受限 | 受限 | 支持 |
| 扩展性 | 差 | 差（有状态） | 好（无状态） |
| 移动端 | App 不友好 | 需 Cookie 支持 | **推荐** |
